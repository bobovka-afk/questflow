# Mini Trello — роадмап геймификации (аватар, прогресс, награды)

Целевой уровень: **готовый продукт** (не «урезанный MVP»): предсказуемое поведение, аудит, эксплуатация и эволюция контента без переписывания ядра.

## 1. Контекст и принципы

### 1.1 Технологический стек

Документ опирается на стек и структуру из [REQUIREMENTS_AND_PLAN.md](./REQUIREMENTS_AND_PLAN.md): **Node.js + TypeScript**, **PostgreSQL**, **Redis**, **REST API**, Docker Compose.

### 1.2 Принцип реализации

- Персонаж (**пол**, **внешность**, **визуальное снаряжение**), предметы и награды — **данные в БД + правила в сервисах** (не обязательная иерархия классов ООП).
- **Тексты в UI** (подписи пресетов, названия предметов, кнопки) — локализация и копирайт; **без сюжета и биографии персонажа** — только выбор пола/внешности/снаряжения.

### 1.3 Архитектура и практики (продуктовый уровень)

- **Границы:** модуль геймификации — отдельный **bounded context** (свои таблицы, сервисы, публичные API); интеграция с карточками через явные точки (application service / domain events), без размазывания SQL по чужим модулям.
- **Источник истины:** прогресс, инвентарь, нормы, HP — **PostgreSQL** с транзакциями и ограничениями целостности; **Redis** — кеш чтения, rate limit, очереди, а не единственное хранилище фактов.
- **Идемпотентность:** все начисления (XP, HP, лутбокс, закрытие периода) — по **стабильным ключам** (`card_id`, `period_id`, `job_run_id`) и уникальным индексам, чтобы повтор джоба или ретрай клиента не дублировал эффект.
- **Асинхронность:** тяжёлые или «в конце дня» сценарии — **очередь** (например BullMQ + Redis) или **outbox** в той же транзакции, что бизнес-событие, с последующей доставкой в воркер — без блокировки HTTP-запроса.
- **Контент:** справочники **пола**, пресетов **внешности**, **визуальных** моделей снаряжения, шаблонов предметов и лутбоксов — **контент-пак** (импорт в БД из JSON/YAML, отдельный пайплайн или админка); обновление набора — через импорт и при необходимости миграции данных в БД; ключи **i18n** в API, строки — из слоя локализации, не захардкоженные в коде правил.
- **Наблюдаемость:** структурированные логи (correlation id), метрики (счётчики начислений, ошибки джоб, длительность транзакций), трассировка; отдельные health/readiness для API и воркеров.
- **Контракт:** **OpenAPI** для всех эндпоинтов геймификации; контрактные/интеграционные тесты на критические потоки; миграции БД — только вперёд или с явным планом отката данных.
- **Безопасность и приватность:** ответы «чужого персонажа» при просмотре профиля (G5) — явный allow-list полей (в т.ч. **HP**, **XP**/уровень — если показываем); без email, норм, инвентаря целиком и прочего приватного; админские правки контента и дропа — отдельная роль и аудит.

### 1.4 Видение продукта (воркфлоу)

- В **профиле** пользователь **создаёт персонажа** (один на аккаунт), выбирая только **пол**, **внешность** (пресеты лица/тела/причёски и т.п. — по продукту) и **визуальное снаряжение** (в т.ч. вид оружия/брони как косметика) — через **справочники** и стабильные `slug`/id в `characters.appearance` (или отдельных полях), чтобы набор опций менялся без правки бизнес-логики.
- У персонажа два явных прогресс-бара: **здоровье (HP)** и **опыт (XP)**.
- **XP** копится от выполненных задач → **уровень** по кривой (таблица/формула в конфиге).
- **HP** в диапазоне **0–100** (потолок при пополнении): если **за календарный день** (в TZ пользователя) **не было** засчитанных выполнений — в конце дня (или по джобу) **снимается** HP; при **выполнении задач** HP **пополняется**, но **не выше 100**.
- **Стрики:** отдельно ведётся серия дней с «хорошим» днём для HP (пополнение / не проиграл день) и серия дней **подряд без выполнений** (стрик «потери» / упадка дисциплины) — для мотивации и будущих наград.
- **Лутбоксы** с предметами для персонажа выдаются за закрытие **дневных** и **недельных** заданий (связка с нормами/квестами периода).
- **Чужой профиль:** пользователь, открыв **профиль другого пользователя**, видит его **персонажа** (пол, внешность, визуальная экипировка), **здоровье (HP)** и **опыт** (как минимум уровень и отображаемый прогресс XP — в том же духе, что и у себя, в рамках политики видимости).
- **Система достижений** — в планах (см. раздел 6); в текущих этапах G1–G6 не раскладывается по задачам.
- Дальше по роадмапу: **экипировка** из лута/наград, затем **просмотр чужих персонажей** (профиль / workspace).

## 2. План геймификации по шагам

### Этап G1: Фундамент данных и API персонажа

**Цель:** один персонаж на пользователя, без начисления XP; фундамент под контент и локализацию.

1. Миграции PostgreSQL: таблица `characters` — `id`, `user_id` (UNIQUE), `name`, **`gender`** (enum или `gender_option_id` со справочником), `created_at`, **`appearance`** (JSONB) со **стабильными ключами** на пресеты внешности; отдельные справочники: пресеты внешности (`character_appearance_presets` или эквивалент), **визуальное снаряжение** (`equipment_visuals`, `weapon_visuals` и т.п.) — `slug`, ссылки на ассеты/превью; без фракций/рас и длинных текстовых описаний персонажа.
2. **Контент-пак с первого релиза:** импорт вариантов **пола**, пресетов **внешности** и **визуального снаряжения** из эталонных JSON (или админ-пайплайн) → БД; в репозитории — эталонные JSON + CI-проверка схемы; сиды в коде только как дев-удобство, в проде — через импорт.
3. REST (маршруты в составе единого API проекта): `GET /characters/me` (текущий персонаж или 404); `POST /characters` (один на пользователя, при существующем — 409); `PATCH /characters/me` (имя, **пол**, **внешность**, **визуальное снаряжение** — политика «только при создании» vs разрешённый рескин по продукту, валидация Zod/Joi). UI — экран/секция **профиля**.
4. Права: только владелец; без привязки к workspace (персонаж глобален для аккаунта).
5. Тесты: unit на сервис создания и уникальность `user_id`; интеграция — создать пользователя → создать персонажа.

### Этап G2: Нормы и начисление XP с карточек

**Цель:** закрытие карточки влияет на прогресс и XP.

6. Модель нормы: `user_norms` (`user_id`, `period_type` `daily` | `weekly`, `target_count`, политика timezone через `users.timezone` IANA); учёт факта за период — **PostgreSQL** (`norm_periods`, счётчики, статусы закрытия периода); Redis — опционально кеш агрегатов для чтения, не источник истины.
7. Правила XP: таблица **`xp_rules`** (или конфиг + миграции при смене баланса) — базовый XP за карточку, бонус за закрытие нормы за период (один раз за период), веса по `list_id`/тегам по мере необходимости; изменения баланса отражаются в данных/миграциях и покрываются тестами.
8. Идемпотентность: карточка не даёт XP повторно; `cards.completed_at` + `card_xp_grants` (`card_id`, `user_id`, `xp_amount`, `granted_at`) или флаг `xp_awarded` на карточке.
9. Интеграция: при PATCH карточки в «done» / установке `completed_at` — транзакция: карточка → грант → счётчик нормы → `characters.total_xp` (см. этап G3).
10. REST: `GET/PATCH` нормы (`/users/me/norm` или `/characters/me/norm`); `GET /characters/me/progress` — период, факт/цель, до бонуса.
11. Rate limiting (Redis): расширить лимиты на частые PATCH карточек (подозрительный гринд).

**Здоровье (HP) и стрики (в связке с G2):**

- Поле персонажа `health` (или `hp`): **целое 0–100**; при пополнении `min(100, health + delta)`.
- **Правило дня:** если за локальный календарный день **не зафиксировано ни одного** засчитанного выполнения (те же события, что и для нормы/XP) — **снять** фиксированное или конфигурируемое количество HP (нижняя граница 0); иначе день считается «активным» для позитивного стрика.
- **Пополнение:** при засчёте выполнения задачи — **добавить** HP по правилу (фикс/процент от недостающего до 100 — на баланс), всегда с потолком 100.
- **Стрики:** `streak_refill` (или аналог) — число **подряд** дней, когда сработало правило «день с выполнениями» / не было штрафа «пустого дня» (уточнить в продукте одним определением); `streak_miss` — число **подряд** дней без выполнений (рост при каждом «пустом» дне, сброс при первом выполнении). Хранение: поля на `characters` + ежедневный идемпотентный джоб или запись в `character_daily_log`.
- REST: расширить `GET /characters/me` полями `health`, `health_max` (100), стрики; `GET /characters/me/daily-summary` (или эквивалент) для календаря/истории дней — для UI и поддержки; ежедневный джоб HP/стриков — в воркере с метриками и алертами на сбои.

### Этап G3: Уровни и кривая опыта

**Цель:** из `total_xp` вычисляется `level`, единая формула для всех.

12. Таблица `level_curve` (`level`, `xp_cumulative`) или функция в коде + юнит-тесты на стык уровней.
13. Поля в `characters`: `total_xp` (BIGINT), `level` (денормализация), опционально `title_id` (звание за уровень).
14. Сервис `applyXp(characterId, delta, source)` в одной транзакции с записью в `xp_ledger` (`id`, `character_id`, `delta`, `source`, `ref_type`, `ref_id`, `created_at`).
15. При новом уровне: опционально `character_level_events` для ленты прогресса (level-up) на фронте; не путать с системой достижений из разд. 6.
16. REST: расширить `GET /characters/me` полями `level`, `total_xp`, `xp_to_next`, `current_level_xp`.

### Этап G4: Экипировка и предметы (данные + слоты)

**Цель:** «шмот» как коллекция; по умолчанию **косметика и звания**; любые бонусы к XP из предметов — только через явные правила в `xp_rules`/конфиге с потолками и аудитом, чтобы не сломать мотивацию.

17. Справочники: `item_templates` (`id`, `slug`, `name`, краткая подпись для UI при необходимости, `slot`, `rarity`, `stats` JSONB, `cosmetic` JSONB); `character_inventory` или `character_equipment` по слотам — только **визуал и игровые слоты**, без повествовательных описаний.
18. **Лутбоксы:** `loot_box_templates` + таблица дропа (веса, редкость, pity/гарантии при необходимости); выдача за **дневные/недельные** цели (связь с G2, идемпотентный грант на период). Открытие: `POST .../open` → транзакция + запись в **ledger открытий**; симуляция дропа покрыта тестами. Доп. источники: уровень, серии норм — те же шаблоны; админка проверки таблиц дропа.
19. REST: `GET /characters/me/inventory`, `POST /characters/me/equip` (слот + item); `GET /characters/me/loot-boxes` (невскрытые).
20. Фронт: слоты и иконки (статика или URL из `cosmetic`); экран открытия лутбокса.

### Этап G5: Просмотр чужого персонажа (профиль, workspace)

**Цель:** авторизованный пользователь при открытии **профиля другого пользователя** (и при необходимости в контексте workspace) получает согласованное представление персонажа с **HP** и **XP**; без отдельного игрового realtime-сервера.

21. **Чтение чужого персонажа для UI профиля:** эндпоинт в составе API профиля/пользователей (например `GET /users/:userId/character` — точный путь как в основном плане маршрутизации) при условии **разрешённой видимости** (например: участники общего workspace, настройка «показывать персонажа», или иная политика продукта — зафиксировать в модуле профиля). В ответе — **имя персонажа**, **пол**, **внешность**, **визуальная экипировка**, **`health` / `health_max`**, **`level`**, **`total_xp`**, при необходимости **`xp_to_next`** / **`current_level_xp`** (тот же смысл, что в `GET /characters/me`, но только разрешённые поля). При отсутствии персонажа или запрете видимости — **404** или **403** по единой политике.
22. **Дополнительно (workspace):** при необходимости узкий эндпоинт `GET /workspaces/:wsId/members/:userId/character` при членстве обоих — тот же **DTO**, чтобы не плодить расхождения.
23. Рейтинг по workspace (топ по `level` / `total_xp`): индексы/материализованное представление или периодический снимок; **кеш Redis** с TTL и инвалидацией при значимых событиях.
24. Социальные реакции («похлопать», эмодзи): лимиты и анти-спам через Redis; аудит при злоупотреблениях.
25. Копирайт UI: короткие **нейтральные** названия предметов и лутбоксов; то же для достижений, когда появится разд. 6 — отдельно от кода.

### Этап G6: Сезоны, уведомления, наблюдаемость, релизная дисциплина

26. **Сезоны:** таблица `seasons`, учёт `season_xp` (или отдельный ledger) для лидербордов «за сезон»; календарь сезонов, награды и тексты в UI согласованы при выкатке.
27. **Уведомления:** события «норма почти выполнена», level-up, лутбокс получен — через **очередь** (BullMQ + Redis) и провайдеры push/email по политике продукта; шаблоны и частота — конфигурируемы.
28. **OpenAPI:** актуальная спецификация эндпоинтов геймификации; при необходимости генерация клиента или проверка контракта в CI.
31. **E2E и нагрузочные сценарии:** персонаж → норма → карточка → XP/уровень/HP/лутбокс; **просмотр чужого профиля** с HP/XP при разрешённой видимости; гонки при параллельных PATCH; регрессия джоба «конец дня».

## 4. Порядок внедрения (рекомендуемый)

**G1 → G3** (персонаж: пол/внешность/визуальное снаряжение, контент-пак справочников, кривая уровней, `applyXp` + ledger) → **G2** (нормы, карточки, **HP и стрики**, воркер закрытия дня) → **G4** (инвентарь, лутбоксы, дроп-тесты) → **G5** → **G6** (сезоны, уведомления, OpenAPI, E2E). **Достижения (разд. 6)** — после ядра событий или параллельно, с тем же стандартом идемпотентности и аудита.

Сначала фиксируются доменная модель, аудит (`xp_ledger`, гранты лута, дневные логи) и контракты API; затем подключаются интеграции с карточками и фоновые джобы без смены схемы «вслепую».

## 5. Риски и ограничения

- **Фарм:** повторное переключение статуса карточки — идемпотентные гранты и серверная валидация.
- **Часовые пояса:** нормы и **срез дня для HP** — одна политика UTC vs `user.timezone`; джоб закрытия дня должен быть идемпотентен.
- **Баланс:** бонусы экипировки к XP — только осознанно и с потолками; по умолчанию косметика; **HP** калибровать по метрикам удержания, не наказывать шумом таймзон и джобов.
- **Операции:** бэкапы, миграции при росте таблиц ledger, ретеншн логов; документированные runbook для повтора «конца дня» и расследования дублей начислений.

## 6. Достижения — **в планах** (не входит в нумерацию G1–G6)

Цель: бейджи/титулы за накопительные и разовые условия (уровень, стрики, объём задач, редкие предметы и т.д.).

- Каталог `achievements`, прогресс пользователя/персонажа, выдача наград (косметика, титул, разовый лутбокс).
- События из существующего ledger XP, дневных логов HP и открытий лутбоксов — как источники триггеров.
- Детализация миграций, API и порядок после стабилизации G2–G4.
