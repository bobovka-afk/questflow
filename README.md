# Questflow — fullstack-приложение

Fullstack‑проект в духе **Trello** с планом **геймификации** поверх командных досок и карточек.

**Бэкенд** — основной контур бизнес‑логики:

- *Платформа и API:* **NestJS**, **TypeScript**, REST, **Swagger**
- *Данные:* **Prisma**, **PostgreSQL**
- *Redis:* кеш, лимиты запросов, refresh-сессии
- *Аутентификация:* **JWT**, **Passport** (Google OAuth, JWT strategy), **bcrypt**
- *Валидация DTO:* **class-validator**, **class-transformer**
- *Почта и логи:* **SendGrid**, **Pino** (nestjs-pino)

**Фронтенд** (сгенерировано AI) — рабочий клиент, не «заглушка»:

- *Клиент:* **React**, **TypeScript**, SPA
- *Сборка:* **Vite**
- *Архитектура:* hybrid (`app/pages/widgets/features/entities/shared`) с alias-импортами (`@app`, `@pages`, `@widgets`, `@features`, `@entities`, `@shared`)

### Frontend структура (hybrid)

- `frontend/src/app` — вход приложения и app-shell.
- `frontend/src/pages` — page-level композиция роутов/экранов.
- `frontend/src/widgets` — крупные UI-блоки страниц.
- `frontend/src/features` — пользовательские сценарии.
- `frontend/src/entities` — доменные сущности (character, quest, board, workspace, invite, reward).
- `frontend/src/shared` — общий API слой и инфраструктурные утилиты.

## Логика приложения: с чего начинается пользователь

### 1. Регистрация и вход (email + пароль)

Регистрация по email и паролю; вход выдаёт **access JWT** и **refresh** в **httpOnly cookie**; по refresh можно обновить access; выход сбрасывает cookie. На чувствительные операции действуют **rate limits**.

### 2. Вход через Google

OAuth2 (**Passport Google**): после колбэка пользователь создаётся или находится, дальше та же схема токенов. Нужны переменные Google OAuth и базовые URL клиента/сервера в `.env`.

### 3. Подтверждение email

Письмо со ссылкой и одноразовым токеном; можно запросить повторную отправку. Пока адрес не подтверждён, часть сценариев может быть ограничена.

### 4. Сброс и смена пароля

Сброс по email: письмо со ссылкой, установка нового пароля по токену. Смена пароля для уже залогиненного пользователя — по текущему и новому паролю.

## Рабочие пространства (workspace)

- **Создание** воркспейса — создатель получает роль **OWNER**.
- **Участники** — таблица членства с ролями **OWNER**, **ADMIN**, **MEMBER**.
- **Приглашения по email**: отправка ссылки/токена, запись `WorkspaceInvite`, письмо через **SendGrid** (настройки в `.env`).
  - Принятие по **id** приглашения со страницы **`/invites`**; либо по **токену** из ссылки.
  - Отклонение приглашённым, отзыв приглашения owner/admin.
- **История активности воркспейса** (аудит на уровне пространства, не досок): фиксируются события вроде создания/обновления воркспейса, приглашений, вступления/выхода/исключения участников, отмены приглашений. Записи хранятся в БД; **просмотр** — только у **OWNER** и **ADMIN**; на клиенте — страница **«Журнал активности»**, переход с экрана участников.
- **Удаление воркспейса** — каскадно удаляет связанные данные (по схеме Prisma).

## Доски, списки, карточки, комментарии

- Иерархия: **Workspace → Board → List → Card**; комментарии привязаны к карточке и автору.
- Операции защищены guards: членство в воркспейсе, для части действий — роли **OWNER/ADMIN**.
- Перенос карточек между списками, позиции, исполнитель, дедлайн, завершение — в соответствующих модулях и DTO.

## Установка и запуск

1. Клонировать репозиторий.
2. Скопировать `backend/.env.example` → `backend/.env` и заполнить переменные.
3. Установить зависимости: `npm install` в `backend/` и в `frontend/` (только для локального запуска без Docker).
4. Применить миграции Prisma (`npx prisma migrate deploy` или `migrate dev`) — только локально; в Docker миграции выполняются при старте контейнера приложения.
5. Запуск API: `cd backend && npm run start:dev`; клиент: `cd frontend && npm run dev` — только локально.
6. Полный стек в Docker: `cd backend && docker compose up --build`.

**Быстрый старт** — шаги **1**, **2** и **6** (без установки зависимостей на машине и без ручных миграций: они собираются и применяются внутри образов).

**Swagger:** `http://localhost:3000/api/docs`

## Идея продукта и вектор развития («что выше карточек»)

Сейчас ядро — **командная работа с задачами** (воркспейсы, доски, карточки). Планируемый следующий слой — **геймификация** поверх выполнения задач: персонаж пользователя, опыт (XP), здоровье (HP), нормы по дням/неделям, стрики, лутбоксы и косметика, прозрачные правила начислений и аудит в PostgreSQL, асинхронные джобы и очереди при росте нагрузки. Видение «над задачами» — мотивация через прогресс и коллекцию, без обязательного сюжета, с упором на данные, идемпотентность и наблюдаемость.
